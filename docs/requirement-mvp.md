## 1. 文档定位

本文档是 `JuneGlass` 当前阶段 **MVP 的唯一总览需求**，覆盖眼镜端与手机端的系统边界、协作关系与验收标准。

### 1.1 本文档回答什么

- 整个 MVP 要解决什么问题、如何闭环
- 眼镜端与手机端各自负责什么
- 两端如何对接、本期做什么、不做什么
- 系统级验收标准

## 2. MVP 核心目标

第一版只解决一个核心问题：

**用户看到当前画面时，按一次唤醒后进入连续对话窗口；眼镜持续提供当前图像与音频，手机在同一会话内持续理解用户提问，并在静音退出前完成一轮或多轮实时图分析与播报。**

### 2.1 系统闭环

```text
眼镜持续采图
  → 用户唤醒（本期按键模拟）
  → 眼镜升档进入连续对话模式，持续传图+音
  → 手机收流 / STT / 有效提问检测
  → 手机按需从最近帧中选图
  → 本地 MiniCPM-V 分析
  → TTS 流式播报
  → 连续静音超阈值后，眼镜回落低档采图
```

### 2.2 系统成功标准

满足以下条件即可视为 MVP 达成：

| # | 标准 |
|---|------|
| 1 | 眼镜 Idle 档可持续向手机推 JPEG 图流，连续运行无明显掉流、崩溃 |
| 2 | 唤醒后眼镜升档同时推图+音，手机可正确拆包与处理 |
| 3 | 手机采集链路不因本地模型推理而中断 |
| 4 | 用户有效提问后，系统可在一次唤醒窗口内触发一轮或多轮分析 |
| 5 | 系统能拿到一张与提问时刻足够接近且足够清晰的图片 |
| 6 | MiniCPM 产出用户可理解的简短答案 |
| 7 | TTS 尽早开始播报，不要求整段生成完毕 |
| 8 | 连续多轮交互不出现明显卡死、掉流、内存失控 |

## 3. 目标场景

### 3.1 支持场景

用户针对**当前看到的画面**发起单轮或连续提问。典型问句：

- 这是什么？
- 前面有什么？
- 帮我看看这个是什么
- 这里写了什么？

### 3.2 不支持场景（第二阶段）

- 帮我找钥匙
- 刚才看到的东西在哪里
- 回看过去几分钟的画面
- 基于历史轨迹继续执行任务

---

## 4. 系统架构

### 4.1 架构原则

| 原则 | 说明 |
|------|------|
| **持续采集** | 眼镜持续供流；手机采集线程常驻收流 |
| **按需分析** | 仅在有效提问后唤醒分析链路 |
| **端侧分工** | 眼镜轻采集+推流；手机重理解+推理 |
| **采集推理解耦** | 禁止在采集链路中执行大模型推理 |

### 4.2 职责矩阵

| 能力 | 眼镜端 | 手机端 |
|------|:------:|:------:|
| 图像采集与 JPEG 编码 | ✓ | |
| 音频采集 | ✓（连续对话 Active 会话） | |
| 唤醒门控 | ✓（本期按键模拟） | |
| 推流 / 收流 | ✓ | ✓ |
| 本地 STT | | ✓ |
| 有效提问检测 | | ✓ |
| 最近帧内存缓存（3–5 帧） | | ✓ |
| 选帧 / 清晰度打分 | | ✓ |
| MiniCPM-V 推理 | | ✓ |
| TTS 流式播报 | | ✓ |
| 深度学习推理 | | ✓ |

### 4.3 执行链（手机端）

手机端采用两条核心执行链，详见 [`phone-requirement-mvp.md`](./phone-requirement-mvp.md)：

1. **采集线程**：常驻收流、轻量处理、维护帧缓存、STT、触发分析
2. **分析线程**：休眠等待有效提问 → 选帧 → MiniCPM → TTS → 再休眠

### 4.4 状态机（眼镜端）

```text
IDLE_STREAM ──(唤醒)──► SESSION_ACTIVE ──(结束/超时)──► IDLE_STREAM
```

| 状态 | 眼镜行为 | 手机侧预期 |
|------|----------|------------|
| `IDLE_STREAM` | QVGA 1fps 仅推 IMAGE | 持续更新帧缓存 |
| `SESSION_ACTIVE` | VGA 2.5fps 持续推 IMAGE + 16kHz PCM，直到静音退出 | 收 WAKE → 持续 STT/选帧/分析 |

详见 [`glass-requirement-mvp.md`](./glass-requirement-mvp.md)。

### 4.5 端到端时序

```text
[Idle]  眼镜 ──IMAGE──► 手机缓存帧（持续）
[唤醒]  眼镜 ──CONTROL_WAKE──► 手机记录 wake_ts
        眼镜 ──IMAGE(升档持续) + AUDIO(持续)──► 手机
[分析]  手机 STT → 有效提问? → 选帧 → MiniCPM → TTS（可多轮）
[结束]  连续静音超阈值 / 超时 / 强制结束
        眼镜 ──CONTROL_SESSION_END──► 手机
        眼镜回落 Idle IMAGE
```

## 5. 本期范围（In Scope）

### 5.1 系统级

| # | 能力 |
|---|------|
| 1 | 眼镜 Idle 档持续 JPEG 图流 |
| 2 | 唤醒后眼镜升档并进入连续对话图+音双流 |
| 3 | 手机持续收流与帧缓存 |
| 4 | 手机本地 STT + 规则化有效提问检测 |
| 5 | 最近 3–5 帧选帧（含清晰度辅助） |
| 6 | 单轮或多轮 MiniCPM-V 图+文分析 |
| 7 | TTS 流式播报 |
| 8 | 眼镜 ↔ 手机 TCP 二进制协议对接 |

### 5.2 眼镜端摘要

| 项 | 本期约定 |
|----|----------|
| 硬件 | XIAO ESP32-S3 Sense |
| 框架 | Arduino |
| 传输 | WiFi STA，TCP Server :8765 |
| Idle | QVGA 320×240，1 fps，仅 IMAGE |
| Active | VGA 640×480，2.5 fps，IMAGE + PCM 16kHz，持续到静音退出 |
| 唤醒 | 板载按键 + 串口 `w`/`e`（模拟） |

→ 展开：[`glass-requirement-mvp.md`](./glass-requirement-mvp.md)

### 5.3 手机端摘要

| 项 | 本期约定 |
|----|----------|
| 采集 | 常驻线程，极轻量单帧处理 |
| 缓存 | 内存 3–5 帧，不落盘 |
| 选帧 | 默认最新帧；模糊则回退；优先唤醒窗口 `[wake_ts-1s, wake_ts+2s]` |
| 触发 | 规则匹配「这是什么」「帮我看看」等 |
| 模型 | 本地 MiniCPM-V，单图+单问 |
| TTS | 流式尽早播报 |

→ 展开：[`phone-requirement-mvp.md`](./phone-requirement-mvp.md)

### 5.4 对接协议摘要

| 项 | 约定 |
|----|------|
| 传输 | TCP，眼镜 Server，端口 8765 |
| 帧类型 | IMAGE / AUDIO / CONTROL_WAKE / CONTROL_SESSION_END |
| 连接 | 单连接混传，小端字节序，14 字节定长头 |

→ 展开：[`transport-mvp.md`](./transport-mvp.md)

## 7. 关键协作约定

以下约定是两端联调必须对齐的「硬接口」，细节见 [`transport-mvp.md`](./transport-mvp.md)。

| 约定 | 说明 |
|------|------|
| 唤醒职责 | 眼镜发 `CONTROL_WAKE`；手机不做唤醒检测（本期） |
| 提问职责 | 手机 STT + 规则判断是否为有效提问 |
| 选帧职责 | 手机在最近帧中选图；眼镜不传多张候选 |
| 时间对齐 | `CONTROL_WAKE.timestamp_ms` 为选帧锚点 |
| 升档间隙 | 眼镜切换分辨率可有 0.3–0.5s 停流，手机用缓存兜底 |
| 会话边界 | `session_id` 关联同一次唤醒的图与音 |

## 8. 性能与稳定性

| 链路 | 要求 |
|------|------|
| 眼镜推流 | Idle 连续 10 分钟无明显掉流；会话升档/回落可重复 10 次不崩溃 |
| 手机采集 | 优先保证不掉流；分析线程不得长时间阻塞采集线程 |
| 手机推理 | 允许重算力，必须与采集解耦 |
| TTS | 播报不影响后续采集 |
| 系统 | 连续多轮交互不明显掉包、卡死 |

## 9. 验收标准

### 9.1 眼镜端单机验收

1. WiFi 连接成功，串口可见 IP
2. TCP 持续收到 Idle `IMAGE` ≥ 5 分钟
3. 按键/串口触发完整会话：WAKE → 升档 IMAGE + AUDIO → SESSION_END → 回落 Idle
4. 连续 10 次会话无崩溃

→ 详见 [`glass-requirement-mvp.md` §11](./glass-requirement-mvp.md)

### 9.2 手机端单机验收

1. 采集线程可持续收流并维护 3–5 帧缓存
2. STT 可转写 Active 会话音频
3. 规则可识别典型提问句
4. 分析线程唤醒后可完成 MiniCPM + TTS，采集不中断

→ 详见 [`phone-requirement-mvp.md`](./phone-requirement-mvp.md)

### 9.3 系统联调验收

1. Idle 期间手机帧缓存持续更新
2. 一次完整交互：唤醒 → 提问 → 选帧 → 分析 → 播报 → 回落
3. 答案与画面内容基本相关，TTS 在可接受延迟内开播
4. 连续 5 轮交互系统稳定

## 10. 风险与假设

| 风险 | 影响 | 缓解 |
|------|------|------|
| 摄像头视角与佩戴视野不一致 | 选帧/分析不准 | 本期以协议打通为主；结构件阶段再优化 |
| 本期无唤醒词，仅按键 | 体验与最终产品有差距 | 不阻塞联调；下期补唤醒词 |
| 手机端栈未定型 | 联调滞后 | 眼镜以协议文档 + Python 脚本自测 |
| 升档短暂停流 | 最新帧可能缺失 | WAKE timestamp + 帧缓存兜底 |
| BLE 未实现 | 功耗偏高 | 本期 WiFi；Idle 压至 1fps |

## 11. 第二阶段扩展（备忘）

以下能力不得混入本期主闭环：

- 找东西任务与任务状态管理

- 眼镜本地唤醒词

## 12. 最终结论

JuneGlass 本期 MVP 的唯一产品闭环：

```text
眼镜 Idle 持续采图
  → 用户唤醒
  → 眼镜升档传图+音
  → 手机 STT 识别有效提问
  → 手机选一张当前可用图
  → 本地 MiniCPM 分析
  → TTS 播报答案
  → 会话结束，回落 Idle
```

设计重点：

| 端 | 重点 |
|----|------|
| 眼镜 | 采集稳定、协议清晰、状态机可靠 |
| 手机 | 采集稳定、触发稳定、选帧可用、推理打通、播报顺畅 |
| 系统 | 两端解耦、接口清晰、可独立开发与分阶段联调 |
